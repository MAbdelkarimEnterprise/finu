Continue the existing FINU website work in this repository.

Project context
- Production reference: https://meetfinu.netlify.app/
- Important deep link: https://meetfinu.netlify.app/#numbers
- Experience reference: https://web.meetcleo.com/
- Stack: Next.js 15 App Router, React 19, TypeScript, Tailwind CSS, Framer Motion, GSAP/ScrollTrigger, Lenis, React Three Fiber, Drei, and Three.js.
- Deployment target: Netlify.

Goal
Evolve FINU from a visually impressive but abstract technology story into a clear, credible, conversion-oriented product website. Preserve FINU's cinematic, AI-native stablecoin identity. Learn from Cleo's product clarity, human storytelling, progressive proof, and contextual calls to action, but do not copy Cleo's layouts, assets, copy, branding, or visual identity.

Before editing
1. Read AGENTS.md, README.md, package.json, and all existing FINU components and styles.
2. Inspect git status and preserve all existing user changes. In particular, determine whether the existing split-type dependency is genuinely useful; keep it only if it improves the implementation, otherwise remove it cleanly.
3. Inspect the two live reference websites at desktop and mobile sizes.
4. Run the current site locally and understand its scroll interactions before changing them.
5. Do not invent customers, testimonials, certifications, regulatory claims, transaction volumes, or performance data. Existing unverified figures should be clearly framed or replaced with truthful capability statements.

Product direction
The homepage must answer these questions quickly:
- Who is Finu for?
- What does Finu enable?
- How does its AI layer improve money movement?
- Why are stablecoin rails useful?
- Why should a visitor trust Finu?
- What should the visitor do next?

Use this positioning direction as a starting point, refining it if the current repository contains stronger approved language:
- Category: AI-native money orchestration
- Audience: businesses, platforms, and treasury teams moving money globally
- Core outcome: Finu interprets intent, selects the appropriate rail, and settles value across markets
- Tone: precise, intelligent, calm, globally ambitious

Implementation requirements

1. Hero
- Retain the distinctive WebGL network atmosphere, but make product meaning more immediate.
- Use one sharp outcome-led H1 and a supporting sentence that names the audience or use case.
- Keep one clear primary CTA and one lower-commitment secondary CTA.
- Make the route/settlement interface cards feel like meaningful product evidence, not decorative labels.
- Ensure the hero remains legible when WebGL is unavailable, reduced motion is enabled, or the device is resource constrained.

2. Product narrative
- Create a coherent progression: problem -> new financial rail -> Finu orchestration -> product outcomes -> proof -> global infrastructure -> CTA.
- Explain "Understand, Route, Settle" with concrete examples and concise copy.
- Show the product in use through interface-led visuals such as routing decisions, settlement status, treasury optimization, and guardrail alerts.
- Avoid vague AI language. Every AI claim should describe a visible action or benefit.

3. Numbers section
- Rebuild #numbers so the deep link lands on a complete, understandable section rather than the middle of a scroll animation.
- Display the key metrics or capabilities in semantic HTML before relying on animation.
- Each number must have a clear label and explanation of why it matters.
- Preserve a premium animated experience, but make scroll-scrubbing progressive enhancement.
- The section must work with keyboard navigation, reduced motion, refresh-at-anchor, and direct linking.

4. Trust and conversion
- Strengthen trust without fabricating proof.
- Clarify what partner/infrastructure logos represent; do not imply partnerships that cannot be substantiated.
- Add appropriate placeholders or structurally complete sections for real proof only when the content is honestly available.
- Add a concise security/infrastructure explanation or FAQ using claims supported by the project.
- Repeat CTAs only where the surrounding content earns them, and use action-specific labels such as "Explore the platform", "See how routing works", or "Open Finu".
- Replace dead "#" links with real destinations or non-clickable elements.

5. Visual system
- Preserve FINU's dark, luminous, infrastructure-grade character.
- Introduce controlled tonal variation so the full page is not one continuous dark visual field.
- Improve hierarchy, body-text readability, contrast, spacing rhythm, and section differentiation.
- Keep the work original. Do not reproduce Cleo's scenic imagery, pill navigation, typography, wording, or compositions.
- Reuse existing design tokens and components where sensible; consolidate repeated values instead of adding scattered one-off styles.

6. Accessibility and resilience
- Keep exactly one meaningful H1.
- Use logical heading order and semantic landmarks.
- Meet WCAG 2.1 AA contrast for essential text and controls.
- Provide visible focus states, keyboard-operable navigation, descriptive link labels, and adequate touch targets.
- Respect prefers-reduced-motion across Framer Motion, GSAP, Lenis, and WebGL.
- Avoid hiding important content inside canvas or animation-only states.

7. Responsive behavior and performance
- Verify at approximately 375px, 768px, 1280px, and 1440px widths.
- Prevent clipped headlines, overlapping cards, horizontal scrolling, and unusable pinned sections.
- Keep expensive WebGL and animation work isolated to client components.
- Avoid unnecessary dependencies and large media assets.
- Maintain good loading behavior and a polished fallback while the 3D scene initializes.

8. Code quality
- Follow the existing architecture and TypeScript conventions.
- Prefer focused reusable components over one oversized homepage component.
- Clean up obsolete code and styles made unnecessary by the redesign.
- Do not modify unrelated files.
- Do not commit, push, deploy, or change external services unless explicitly asked.

Acceptance criteria
- The hero communicates audience, product, and outcome within a few seconds.
- The homepage has a clear narrative and no dead CTA links.
- /#numbers works correctly when opened directly and remains understandable without animation.
- The experience is original and recognizably FINU, not a Cleo clone.
- Important content remains accessible with JavaScript animation reduced or WebGL unavailable.
- Desktop and mobile layouts are polished.
- Browser console shows no application errors.
- npm run build succeeds with zero TypeScript or compilation errors.
- Run any available linting or tests and report their results.

Working approach
- Implement the work rather than only writing a plan.
- Make informed decisions from the existing code and references.
- Keep me updated briefly while working.
- After implementation, inspect the result in a real browser at desktop and mobile sizes and iterate on visible issues.
- Finish with a concise summary of changes, files touched, verification performed, and any remaining content decisions that require real business information.
