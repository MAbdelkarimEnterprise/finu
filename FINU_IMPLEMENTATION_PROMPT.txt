WHAT IS CURRENTLY MISSING

The site is visually competent, but it does not yet feel alive enough:

1. The WebGL hero shader is so dark and heavily covered by the hero veil that it reads as an almost static navy gradient.
2. Pointer interaction is too subtle and may not reliably activate when content overlays the canvas.
3. The shader exists only in the hero. Below the fold, the page becomes one long, nearly uniform dark field.
4. Cards use a fixed CSS hover rotation rather than cursor-aware light, depth, or parallax.
5. Product interface demonstrations are clear but mostly static after entering the viewport.
6. Section transitions lack energy and visual continuity.
7. Partner logos are too dim on mobile and the marquees can feel sparse.
8. The metrics count-up can temporarily display awkward values such as “0M+” while entering the viewport.
9. The README and previous implementation prompt describe an older architecture and need to be brought in line with the real codebase.

IMPLEMENTATION REQUIREMENTS

1. UPGRADE THE HERO SHADER

Improve the existing shader in src/components/MidnightIntelligenceShader.tsx instead of replacing it with a generic gradient or importing a large 3D library.

The upgraded shader should include:

- Broad, slow domain-warped FBM fields for dimensional blue and violet movement.
- Thin data-like filaments or contour lines that travel through the field.
- A few restrained cyan energy pulses that move along those filaments.
- A subtle depth layer so the background feels spatial rather than flat.
- A soft pointer field that bends or attracts nearby flow.
- Gentle brightness and distortion changes based on pointer velocity.
- A very slow ambient “breathing” intensity when the pointer is idle.
- Fine grain to prevent banding.
- A vignette and content-safe dark zone behind the hero text.

Make the shader visibly active without making it fast or noisy:

- Broad color cycles: approximately 16–24 seconds.
- Traveling data pulses: approximately 5–9 seconds.
- Pointer movement must be damped and elegant.
- No rapid flashes, rainbow gradients, smoke, fire, liquid-metal effects, or full-screen brightness changes.

The current shader colors should remain rooted in:

- Base: #050816
- Deep field: #101B49
- Electric blue: #3A43B7
- Violet: #7B4DFF
- Cyan pulse: #22C7E8

Bind pointer tracking to the hero container or a reliable parent rather than depending exclusively on receiving pointer events directly on the canvas behind the content.

Add robust shader lifecycle handling:

- Pause animation while the document is hidden.
- Pause when the shader is well outside the viewport.
- Cap device pixel ratio.
- Reduce complexity on mobile.
- Disable pointer behavior on touch devices.
- Respect prefers-reduced-motion with a polished static fallback.
- Keep the DOM hero interface fully usable if WebGL fails.
- Clean up programs, shaders, buffers, observers, and listeners.

Reduce the opacity of the current hero veil where necessary so the shader is actually visible, but maintain excellent headline contrast. The right side around the product UI can be more luminous than the text side.

Do not add Three.js or React Three Fiber unless there is a genuinely unavoidable technical reason. The current plain-WebGL approach is sufficient.

2. ADD CONTROLLED AMBIENT MOTION BELOW THE HERO

The page needs visual continuity after the hero, but do not place a heavy WebGL canvas behind every section.

Create a lightweight reusable ambient system using one of these approaches:

- One additional low-cost shader canvas that is reused across the products/metrics region and pauses when not visible; or
- CSS/SVG animated light fields and data paths derived from the hero shader language.

Use a maximum of two active WebGL canvases on the page at once.

Add restrained section-level details:

- Slow blue/violet light fields that drift behind section boundaries.
- Thin SVG data paths that connect selected sections.
- Small pulses that travel through paths when a section enters the viewport.
- A soft grid or coordinate texture in one or two places only.
- Gradient separators that appear to transmit data from one section to the next.
- Distinct tonal treatments for cards, products, trust, community, and CTA sections.

The page should no longer look like one uninterrupted navy rectangle, but it must still feel like one coherent system.

3. MAKE CARDS REACT TO THE USER

Replace the fixed one-direction card rotation with a reusable cursor-aware interaction component or hook.

For desktop pointer devices:

- Calculate pointer position relative to each interactive card.
- Apply no more than approximately 1.5–2 degrees of rotation.
- Move the internal highlight independently from the card.
- Expose CSS variables such as --pointer-x and --pointer-y for radial edge lighting.
- Add a restrained electric-blue/cyan border bloom near the pointer.
- Smoothly return cards to rest on pointer leave.

Disable tilt on touch devices and when reduced motion is enabled.

Do not make text, buttons, QR codes, or form-like controls difficult to click.

Apply the effect selectively to major cards, not every small row.
